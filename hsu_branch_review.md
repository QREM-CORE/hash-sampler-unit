# Branch Review: `hsu_change`

Review of 6 commits on `hsu_change` vs `main` (base: `0982701`).

---

## A. Overall verdict

**Mostly good but incomplete.**

The RTL changes are on-target and faithfully implement the approved plan. The architectural boundaries are preserved. However, one of the three original bugs is only *half* fixed — the σ capture logic is correct, but the `seed_wr_beat_cnt` output counter that caused the original ρ-overwrite still rolls over. The test infrastructure is also broken: the `mlkem-python` submodule is not initialized, the old `test_vectors.json` was deleted, and no new vectors exist on disk. You cannot run the testbench as-is.

---

## B. What it fixed correctly

- **5th-beat coordinate injection (Change 1b):** Implemented exactly as planned. `coord_beat_pending` sets on the 4th seed beat, then a synthetic 5th beat injects `{48'b0, row_lat, col_lat}` with `keep=0x03` and `t_last=1`. The `t_last` suppression on beat 4 is correctly gated on `hsu_mode_i != MODE_SAMPLE_NTT`. This fixes the matrix generation bug for both KeyGen and Encaps.

- **`row_i`, `col_i`, `cbd_n_i` ports (Change 1a):** All three 8-bit sideband ports added in the correct location (lines 68–70). Latched on `start_i` via `row_lat`, `col_lat`, `cbd_n_lat`. Clean.

- **σ register capture (Change 1c):** `sigma_reg` (256-bit), `sha512_beat_cnt` (3-bit), and `sigma_valid` are correctly implemented. Beats 4–7 of SHA3-512 output are trapped locally. `seed_req_o` and `seed_we_o` are correctly forced to 0 during those beats, preventing Seed RAM writes. The `keccak_t_ready_i = 1'b1` unconditional accept for captured beats is correct.

- **CBD σ||N feed (Change 1d):** `sigma_feeding` FSM correctly streams 5 beats (σ[0–3] + N byte) with proper `t_last` on beat 4 and `keep=0x01`. Counter is 3 bits. Backpressure via `keccak_t_ready_o` is respected.

- **MUX priority ordering:** The `default` case of the `input_sel_i` mux correctly prioritizes `sigma_feeding` > `coord_beat_pending` > normal seed path. No ambiguity.

- **Testbench wiring:** `row_i`, `col_i`, `cbd_n_i` connected. Config parsing for `ROW`, `COL`, `CBD_N`, `RUN_G_FIRST` added. Seed feeder now waits for `keccak_t_ready_o` (backpressure-aware). `RUN_G_FIRST` flow correctly runs a SHA3-512 phase before the target test.

- **Verification infrastructure:** Makefile updated to use `mlkem-python` reference implementation for vector generation. `generate_tb_files.py` updated to emit new config keys.

- **Architectural boundaries preserved:** No $k$-loops added to HSU. Controller ownership of sequencing untouched. Row buffer and memory map unchanged. Package file unchanged.

## C. What it missed

- **`seed_wr_beat_cnt` rollover is NOT fixed.** This is the original Bug 2 from our review. Look at line 445:
  ```sv
  logic [$clog2(SEED_BEATS)-1:0] seed_wr_beat_cnt;
  ```
  `SEED_BEATS` is 4, so this is still a 2-bit counter. During SHA3-512, the σ capture logic (Change 1c) correctly suppresses `seed_req_o` on beats 4–7, so σ *won't overwrite ρ in Seed RAM*. **However**, `seed_wr_beat_cnt` still freely increments on `seed_req_o && seed_ready_i` and will roll over on any future mode that writes more than 4 beats. The counter itself is not gated. For the current SHA3-512 path this is *accidentally safe* because `seed_req_o = 0` on beats 4–7 prevents the counter from advancing past 3. But it's fragile — if anyone ever adds a new mode that writes >4 beats without the same guard, the rollover will silently corrupt data. **Not blocking for this branch, but should be documented as a known fragility.**

- **`mlkem-python` submodule not initialized.** `git submodule status` shows `-ab701006...` (the `-` prefix means uninitialized). The `verif/mlkem-python/` directory is empty. The Makefile's `$(VECTOR_STAMP)` target depends on `verif/mlkem-python/tests/Intermediate_hash_sampling.py`, which doesn't exist. **`make build` or `make run_hash_sampler_unit_tb` will fail.** The old hardcoded `test_vectors.json` (645 lines) was deleted from the repo. There are currently zero test vectors on disk.

- **No new test vectors committed.** The plan specified Tests A2 (asymmetric coordinates), D (SHA3-512 baseline), and G (G(d)→CBD flow). None of these exist as committed JSON entries. They presumably would be auto-generated by `Intermediate_hash_sampling.py`, but since the submodule is empty, this is untestable.

- **No `sigma_valid` assertion/check in TB.** The TB reads `DUT.sigma_valid` and prints `DUT.sigma_reg` via `$display`, but does **not** compare it against expected values. There is no pass/fail check for σ correctness. This is a monitoring stub, not a verification check.

## D. What looks wrong or risky

### D1. `seed_wr_beat_cnt` still 2-bit (Low risk, not blocking)
As noted above. The σ capture guard makes this safe *today* for SHA3-512, but it's a latent bug for any future mode extension. **Follow-up item.**

### D2. `sigma_reg` slice indexing uses synthesis-unfriendly pattern (Medium risk)
Line 247:
```sv
sigma_reg[64*(sha512_beat_cnt - 3'd4) +: 64] <= keccak_t_data_o;
```
Line 329:
```sv
keccak_t_data_i  = sigma_reg[64*sigma_feed_cnt[1:0] +: 64];
```
These are variable-index part-selects with multiplication. Most synthesis tools handle `+:` with constant width and variable base, but the `64 * (counter - offset)` multiplication may produce warnings or suboptimal logic on some FPGA toolchains. The designer should verify this synthesizes cleanly. If not, replace with a simple `case` statement on the counter value. **Not blocking, but verify during synthesis.**

### D3. `sigma_valid` cleared on every `start_i` (Potential concern)
Line 241–242:
```sv
if (start_i) begin
    sha512_beat_cnt <= '0;
    sigma_valid     <= 1'b0;
```
Every `start_i` pulse clears `sigma_valid`, regardless of mode. This means if the controller pulses `start_i` for an NTT matrix generation request *after* σ was captured, `sigma_valid` goes low. The `sigma_reg` contents are preserved (not zeroed), so the CBD feed will still use the correct data. But any external logic that checks `sigma_valid` as a guard will see it as invalid mid-KeyGen. **Not functionally broken, but potentially confusing for the controller designer.** Consider only clearing `sigma_valid` when `hsu_mode_i == MODE_HASH_SHA3_512`.

### D4. `coord_beat_pending` gated only on `MODE_SAMPLE_NTT` (Correct but fragile)
Line 219:
```sv
end else if (hsu_mode_i == MODE_SAMPLE_NTT) begin
```
If a future mode also needs coordinate injection (unlikely but possible), this gate would need updating. Acceptable for now, but flagging it.

### D5. `input_sel_i` value `2'b00` used for seed beat counting (Matches current RTL)
Line 198:
```sv
else if (input_sel_i == 2'b00 && seed_rvalid_i && keccak_t_ready_o)
```
This is consistent with the current `input_sel_i` mux where `default` (including `2'b00`) is the seed path. Correct.

## F. Minimal next-step recommendation

**Accept with cleanup items before merging.**

The RTL changes are correct and on-target. The branch should not be merged until:

### Blocking (must fix)
1. **Initialize the `mlkem-python` submodule** and verify `make` can generate vectors end-to-end. Without this, the branch has zero test coverage.
2. **Add at least one committed test vector** that can be run without the Python reference (a hardcoded fallback), so CI doesn't silently pass with no tests.

### Non-blocking (should fix soon)
3. **Add a `$error` assertion** in the TB that compares `DUT.sigma_reg` against expected bytes, not just `$display`.
4. **Guard `sigma_valid` clearing** — only clear on `start_i && hsu_mode_i == MODE_HASH_SHA3_512`, not on every `start_i`.
5. **Verify variable part-select synthesis** — run a quick `make synth` to confirm `sigma_reg[64*cnt +: 64]` doesn't produce synthesis warnings.

### Follow-up (can defer)
6. Document that `seed_wr_beat_cnt` is still 2-bit and relies on the σ capture guard to prevent rollover during SHA3-512.
7. Add `MODE_HASH_SHA3_256` regression test to confirm 5th-beat injection stays dormant.
